# 协作护栏

本文件定义本工作区内的强制协作规则。Kiro 在任何实现、修改、调试、重构、测试建议、Git 操作和发布相关任务中，都必须优先遵守这些规则。

## 1. Git 权限边界

- 未经用户明确授权，禁止执行任何 `git commit`。
- 未经用户明确授权，禁止执行任何 `git push`、`git push --force`、标签推送或其他会修改远端仓库状态的操作。
- “明确授权”必须来自当前对话中的直接批准，含义清晰，不能靠默认推断、上下文猜测或“用户之前可能同意过”来代替。

## 2. 高风险操作二次确认

涉及以下操作时，必须先暂停并进行二次确认，说明将执行什么、影响范围是什么、是否可回退，然后等待用户明确回复：

- 删除：删除文件、目录、数据、记录、资源
- 回滚：`git revert`、`git reset`、回退版本、回退数据
- 迁移：数据库迁移、结构变更、批量数据修复、初始化重建
- 发布：部署、上线、发版、推送到生产或共享环境

没有二次确认，不得执行。

## 3. 默认执行方式

- 可以先完成分析、改代码、写补丁、准备命令、整理提交说明。
- 走到 `commit`、`push`、删除、回滚、迁移、发布这些步骤前，必须停下来请求确认。
- 如果用户的表述不够清晰，必须按“未授权”处理，而不是自行放宽解释。

## 4. 确认用语标准

只有当用户明确表达以下含义时，才视为授权：

- “可以提交”
- “可以推送”
- “现在执行迁移”
- “确认删除”
- “确认回滚”
- “确认发布”

以下情况一律不视为授权：

- 用户只是在讨论方案
- 用户只是在询问是否能做
- 用户表达了目标，但没有明确批准执行高风险动作

## 5. 回复风格要求

当需要确认时，Kiro 应使用简洁、直接、可执行的提示语，例如：

- “代码修改已完成，是否现在执行 `git commit`？”
- “本地提交已准备好，是否允许我推送到 GitHub？”
- “这一步会执行数据库迁移并修改现有表结构，是否确认继续？”

不要把高风险动作包装成已经默认会执行的下一步。
